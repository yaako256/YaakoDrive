# YaakoDrive設計メモ(ブレスト段階)
YaakoDriveは個人(自分&友人&家族)用のクラウドである。
プロジェクト名は、個人用であることをわかりやすくするため、
Yaako(自分の活動名で個人用である意) + Drive(クラウドストレージの意)
とし、プロジェクト内容がわかりやすい名前とする。
GoogleDriveなど、世の中にはいろいろなクラウドがある。
自宅サーバを利用して、個人(自分&友人&家族)用のクラウドを作ろうというプロジェクトである。

# 採用言語について
フロントエンド: React (PWAとしても提供)(PCなどではブラウザから使用)
バックエンド: Rust

Dockerを使用する。
本番データはbind mountでコンテナ外から参照可能にする。
保存先ホストパス: /home/yaako/yaakodrive-data/data:data
開発中保存パス: - ./dev-data:/app/data
(プロジェクトコードとデータは別ディレクトリで管理する)

# 開発について
ユーザは以下の方法でYaakoDriveを利用する。
1. ブラウザ (PC・スマホ共通)
2. PWA (スマホのホーム画面に追加してアプリ風に使用)

# このシステムで挑戦したいこと
## sharedクレートについて
今まではsharedクレートを作り、それをほかのほぼすべてのクレートに依存させていた。
しかし、今回はインターフェースなどの概念を考え、`shared`クレートに重みをもたせすぎない設計を心掛ける。

## error形について
今までは`shared`クレートに`errors.rs`を置いて`AppResult`と`AppError`、それを各workspaceに依存させていた。  
しかし、このプロジェクトは実験的にそれをやめてみようと思う。  
各クレートに`error.rs`を作り、そこに、各クレート専用とError型とResult型を作成する。
ユースケースを定義するappクレートでは、他クレートのError型をまとめた型である`AppError`型を作り、それを使った型である`AppResult`型を戻り値とする。

## ユーザ判別について
自前でJWT認証を実装し、安全性を担保する。(詳細は認証方式セクション参照)


# ネットワーク設計
- アクセス手段: Tailscaleのみ
- Tailscaleで対応できなくなった場合は今後拡張する
- ドメインは一旦考えない。自前DNS構築時に検討する。

# 認証方式
- JWTベースの認証システムを実装する
- パスワードハッシュ: argon2
- JWT発行・検証: jsonwebtoken クレート
- Access Token + Refresh Token方式を採用する
- Access Tokenは短寿命
- Refresh Tokenは長寿命
- Access Token失効時はRefresh Tokenにより再発行する


# バックアップ設計
- 本番データはbind mountでコンテナ外から参照可能にする
- 保存先: /srv/yaakodrive/data など明示的なホストパス
- バックアップ先①: 外付けHDD (同建物)
- バックアップ先②: 別PC or 別クラウド (オフサイト)
- ツール: restic または rclone を検討

# バックアップ設計についてのメモ
ツールは基本はホスト側にインストールをするため、一応ホストを汚してしまう。
しかし、YaakoDriveとは完全に別物であるため、ホストを汚す云々は一旦考えない。
ホストを汚したくない場合、バックアップ専用コンテナを立てるという方法がある。
バックアップ専用コンテナ(cronを含む)を作り、yaakodrive-dataを読み取り専用マウント、cronで定期実行という方法が丸そうである。
また、年に数回程度、テスト環境への復元試験を実施することで、安全性も確認する。

# ファイルの保存構造
ディスク上はフラットに置き、構造はDBだけで管理する構造とする。
```
構造例
yaakodrive-data/
  └── files/
        ├── a3f8c2d1.jpg  ← UUIDなどで管理
        └── b9e1f4a2.pdf
```
これにより、ファイル移動・リネームなどの操作を最小限にに、構造変更をDB更新だけで完結するようにしている。
また、バックアップは以下のように行う
```
【このファイル構造】
バックアップ対象が2つになるだけ

1. ファイル本体
  /home/yaako/yaakodrive-data/files/
    ├── a3f8c2d1.jpg
    └── b9e1f4a2.pdf
  → resticでそのままバックアップ

2. DBのデータ (PostgreSQL)
  → pg_dumpで定期エクスポートしてバックアップ
```
復元するときは「ファイル本体を戻す」＋「DBを戻す」の両方をやれば元通りになる。


# ストレージ設計
物理ファイルはUUIDで保存する。

例
files/
 ├ 550e8400-e29b.dat
 ├ 550e8400-e29c.dat

ユーザに見せるファイル名はDBで管理する。
同一ファイル名の判定は同一フォルダ内のみ行う。

# ファイル名重複ルール
同一フォルダ内で同名ファイルが存在する場合、
アップロード時に警告を表示する。
ユーザが続行を選択した場合、
```

↓
example (1).jpg
example (2).jpg
```
のように自動リネームして保存する。

# DB設計
## 採用DB
- PostgreSQL
- 理由
  - 練習も兼ねて本格的なDBを利用したい
  - SQLiteより将来的な拡張性が高い
  - 友人が利用しており興味があった

## 削除方式
削除は論理削除(deleted_at)で管理する。
例
- deleted_at IS NULL
  → 通常表示
- deleted_at IS NOT NULL
  → ゴミ箱表示
一定期間経過後に物理削除する。

## テーブル構成(予定)
### users
ユーザ管理テーブル。
保持例
- ユーザ名
- パスワードハッシュ
- 権限
- ストレージ上限

---
### refresh_tokens
Refresh Token管理テーブル。
保持例
- ユーザID
- トークン
- 有効期限

---
### nodes
ファイル・フォルダの論理構造を管理するテーブル。
ファイル本体の情報は持たない。
保持例
- オーナーユーザID
- 親ノードID(parent_id)
- 名前
- ノード種別(file / folder)
- deleted_at

フォルダ構造は parent_id により表現する。
例
MyDrive
 └ 写真
     └ cat.jpg

---
### file_contents
ファイル実体に関する情報を管理するテーブル。
フォルダは保持しない。
保持例
- node_id
- 保存ファイル名(UUID等)
- MIME Type
- ファイルサイズ

実際の保存ファイルとnodesを紐付ける役割を持つ。

---

フォルダ移動やリネームはDB更新のみで完結する。
